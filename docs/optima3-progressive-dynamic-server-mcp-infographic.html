Modules are a flat list of dot-paths (pms.reservation.booking); parent/child is inferred from the prefix. Explore accepts a single path, an array (bulk), or a wildcard — pms.* expands a whole subtree, * the entire platform. Everything is role-filtered.

Two Execution Paths

Invoke is primary; native tool promotion is a later phase
Primary · build now

Path B — Invoke (stateless)

A universal dispatcher runs any catalog function by name — no server state, no dynamic tool registration. Matches the DMZ stateless HTTP transport.

  • Single: { name, args }
  • Bulk: { calls:[…] } — parallel-safe reads run via Task.WhenAll, writes run in order
  • Args validated against the function's JSON schema
  • Each call returns its own {success,data} / {success,error} — a batch never fails wholesale
Later phase

Path A — LoadTools (stateful)

Promote selected functions to native MCP tools for repeated use. Deferred — hard on the current stack.

  • Needs per-token dynamic tool lists
  • C# SDK uses assembly-scanned tools + stateless HTTP → requires Stateless=false sessions
  • Would change the shared Silverbyte.Common.Net wiring, not just DMZ
  • Stub returns "use Invoke" for v1

Cross-Branch Workflow Example

One task, functions gathered from different branches of the tree
1
Explore pms.* and crm.* — discover reservation + guest functions in a single bulk call.
2
DescribeTool ["CRM_GetGuests","PMS_GetReservations"] — pull just those two schemas.
3
Invoke (bulk, parallel-safe) CRM_GetGuests + PMS_GetReservations — both reads run concurrently.
4
Invoke a write, e.g. PMS_CreateReservation — runs sequentially, honoring requiresConfirmation.

The agent assembled a booking workflow from the pms and crm branches without ever loading the other ~2,490 tools.

Auth & Request Context

Reuse what BaseTool already does — no AsyncLocal needed
// The SDK builds each tool per request with IHttpContextAccessor —
// exactly the pattern MCP/BaseTool.cs already uses today.
var ctx = new OperationContext {
    Role  = http.User.Claims...,        // role from the JWT ClaimsPrincipal
    Token = bearerFromAuthorizationHdr, // forwarded onto every *APIClient.Token
    Http  = http
};
// Explore / DescribeTool / Invoke all filter by role;
// Invoke re-checks role per call (defense in depth).

The TS prototype needed AsyncLocalStorage<{role,token}>. On .NET, HttpContext gives us the same request-scoped context for free — the existing BaseTool already reads the bearer this way.

Building the Catalog

Hybrid — generate the bulk, curate what matters
Generated

Reflect over NSwag *APIClients

  • Enumerate public *Async methods on PMS/CRM/ORG/SHR/FIN/BI clients
  • Name = "{MODULE}_{Method}"; module inferred from client + method name
  • Get*/Search*/List* → read, else write
  • Input schema derived from parameter types; start with simple-param methods
Curated

Hand-written descriptors win

  • The existing 4 tools become curated catalog entries
  • Better titles/descriptions, requiresConfirmation on destructive ops
  • Curated entries override generated ones of the same name
  • Grow curated coverage module-by-module over time
Browser / Optimon: the React side proxies to the same backend catalog (via src/api/services/agent.js), and Optimon's OpenAI tool loop calls it in-process. One catalog, one source of truth — no schemas duplicated in JS.
