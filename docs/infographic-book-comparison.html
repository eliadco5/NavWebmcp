How the Approaches Differ

3-Call MCP (agent-driven)
The agent calls three separate tools sequentially. It must parse the slot list, extract a slotId, decide to validate, and carry all prior results in its context window for each subsequent call.
→ searchAvailability({date, partySize})
  [agent reasons: pick slotId from response]
→ createReservation({slotId, name, partySize})
  [agent reasons: now validate]
→ getReservation({reservationId})
book() — frontend-orchestrated
Business logic lives in the browser page. The agent calls one tool — book() — registered in document.modelContext. The page handles orchestration; the agent receives a validated result.
→ book({date, time, partySize, name})
  [browser: search → create → validate]
← {reservation, validated: true}
Why context tokens are 0 for book(): In a real LLM session, every prior tool result accumulates in the context window. book() makes a single call with no prior tool exchange to carry.
