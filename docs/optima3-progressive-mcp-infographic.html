Silverbyte.Optima.Cloud · MCP Architecture

Expose the App's Own Functions
to an Automation Agent — via One URL

The same business function a user triggers by clicking "Create reservation" is exposed to an external agent through a single stateless MCP URL. No browser, no navigation. Progressive tool disclosure keeps thousands of functions from ever flooding the agent's context.

External automation agent One stateless URL · /mcp Catalog = BFF controller actions .NET 8 · Optima3.DMZ → Optima3.Application

What we expose

  • BFF controller actions in Silverbyte.Optima3.Application — exactly what the React src/api/services/*.js functions call

  • Reached over one stateless URL by an external automation agent
This is NOT

  • ❌ Wrapping the raw PMS/CRM microservice clients — they miss BFF orchestration
  • ❌ Any navigation, mounting, or human-in-the-loop

Modules are a flat list of dot-paths (pms.reservation); parent/child inferred from the prefix. Explore takes a path, an array (bulk), or a wildcard — pms.* a subtree, * the whole platform. All role-filtered.

Execution Paths

Invoke is primary; native tool promotion is a later phase
Primary · build now

Invoke (stateless)

A universal dispatcher runs any catalog function by name → forwards to the BFF action. No server state. Matches the stateless URL model.

  • Single { name, args } or bulk { calls:[…] }
  • Parallel-safe reads run via Task.WhenAll; writes run in order
  • Args validated against the function's JSON schema
  • Per-call {success,data}/{success,error} — a batch never fails wholesale
Later phase

LoadTools (stateful)

Promote functions to native MCP tools for repeated use. Deferred — hard on this stack.

  • Needs per-token dynamic tool lists
  • C# SDK: assembly-scanned tools + stateless HTTP → needs Stateless=false sessions
  • Touches shared Silverbyte.Common.Net, not just DMZ
  • v1 stub returns "use Invoke"

Building the Catalog

Hybrid — generate the bulk from BFF actions, curate what matters
Generated

From the BFF action surface

  • Reflect over Application.Client methods (or the BFF controller types)
  • Name = "{Controller}_{Action}"; module from controller folder/namespace
  • [HttpGet]/Get*/Search*/List* → read, else write
  • Input schema from the action's request DTO; start with simple signatures
Curated

Hand-written wins on name clash

  • Better titles/descriptions for high-value actions
  • requiresConfirmation on destructive ops (cancel, delete)
  • The existing 4 DMZ tools fold in here
  • Grow curated coverage controller-by-controller

Example: an automation run

Functions gathered from different branches, all over the URL
1

2

4

Definition of Done

How we verify the port
✓
Build & run DMZ with MCPServer:Enable=true and Services:Application → the BFF.
✓
Small root list: Inspector on /mcp shows only Explore / DescribeTool / Invoke.
✓
Navigation works: manifest → module → functions; bulk + wildcard pms.*.
✓
Invoke reaches the BFF: orchestration runs (e.g. Reservation_Search free-text does the guest lookup); JWT forwarded.
✓
Parity with the UI: a real button and the agent hit the same BFF action → same result.
✓
RBAC: filtered tree; forbidden → FORBIDDEN; bad args → INVALID_ARGS (that call only).
Watch out — reverse dependency. The BFF already references a DMZAPIClient (BFF → DMZ). So DMZ must reach the BFF by HTTP (a generated Application.Client or HttpClient), never a project reference back — that would create a build cycle.
